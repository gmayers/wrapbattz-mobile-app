# TOOLTRAQ Rebrand — Phase 1 Design

**Date:** 2026-04-20
**Scope:** Phase 1 of a two-phase redesign. Phase 2 (role-based navigation restructure, center Scan button → QuickActionModal) gets its own spec/plan cycle later.
**Branch:** `feature/tooltraq-rebrand` off `master`.

## Goals

1. Fix two user-reported bugs that block workflow.
2. Rebrand the app from BattWrapz to TOOLTRAQ — new name, new bundle identity, new color palette, new in-app logo. Effectively launch a fresh app identity (new slug, new bundle ID, version reset to 1.0.0).

## Non-goals

- Role-based navigation restructure (Phase 2).
- Center Scan button / quick-action flow redesign (Phase 2).
- Backend rename. The API host `webportal.battwrapz.com` is unchanged.
- Square app icon + splash screen asset swap. Existing `battwrapz` icon/splash remain until square logo variants are produced.
- Terminology rename (Devices → Tools, etc.).
- Sentry project rename.

## Design

### Bug A — Submit Report button clipped behind tab bar

**Where:** `src/screens/CreateReportScreen.js:757` — Submit button sits in a plain bottom `View` inside the ScrollView with no bottom padding compensating for the OS navigation / home-indicator area. (Note: `CreateReport` is pushed on top of `MainTabs` as a Stack.Screen, so the app's tab bar is hidden — the actual culprit in Image #3 is the Android 3-button system nav bar, not the app tabs.)

**Fix:** add `paddingBottom: insets.bottom + 16` to the ScrollView's `contentContainerStyle`, pulling `insets` from `useSafeAreaInsets()`. `16` is breathing room above the home-indicator / system-nav area. No structural change — button stays in flow.

**Acceptance:** Submit Report button is fully visible above the OS system navigation / home indicator on Android 3-button nav, Android gesture nav, iOS home indicator, and iOS older home-button devices. Success alert dialog is not obscured.

### Bug B — NFC "Writing…" text misleading

**Where:** `src/screens/AddDeviceScreen.js:1104` — during NFC write, button shows `"Writing..."` which reads like a background operation is happening, but the user actually needs to physically hold the NFC tag to the device.

**Fix:** text swap only.

| File | Before | After |
|---|---|---|
| `src/screens/AddDeviceScreen.js:1104` | `title={isWritingNfc ? 'Writing...' : 'Write Data to Tag'}` | `title={isWritingNfc ? 'Hold tag to device…' : 'Write Data to Tag'}` |
| `src/screens/QuickAction/QuickActionModalScreen.tsx:349` | `title={upgrading ? 'Upgrading…' : 'Upgrade NFC tag'}` | `title={upgrading ? 'Hold tag to device…' : 'Upgrade NFC tag'}` |
| Any equivalent in NFCManager tabs (EditTab, WriteTab, FormatTab, LockTab, UnlockTab) | "…ing" text | `'Hold tag to device…'` |

**Acceptance:** when the user presses any NFC write/upgrade/format/lock/unlock action, the button label tells them to present the tag, not that something is loading.

### Rebrand — color palette

**Approach:** same yellow primary in both light and dark themes. Dark mode is the polished / mockup-aligned look; light mode remains functional.

**Changes in `src/context/ThemeContext.js`:**

`lightTheme` block:
```js
primary: '#FFC72C',         // was #FF9500
primaryDark: '#E5AE18',     // was #FF8C00 (pressed state)
primaryLight: '#FFF4D1',    // was #FFF3E0
primaryTint10: 'rgba(255, 199, 44, 0.10)',
primaryTint6:  'rgba(255, 199, 44, 0.06)',
primaryTint5:  'rgba(255, 199, 44, 0.05)',
onPrimary: '#0F1722',       // NEW — near-black, for text on yellow backgrounds
```

`darkTheme` block:
```js
primary: '#FFC72C',         // was #FF7700
primaryDark: '#E5AE18',     // was #FF9500
primaryLight: '#3D2F08',    // was #3D2600
primaryTint10: 'rgba(255, 199, 44, 0.10)',
primaryTint6:  'rgba(255, 199, 44, 0.06)',
primaryTint5:  'rgba(255, 199, 44, 0.05)',
onPrimary: '#0F1722',       // NEW — dark text reads on yellow in both modes
```

**New token — `onPrimary`:** text color to use on top of `primary` surfaces. Both themes set it to `#0F1722` because yellow is light in both modes.

**Button component update** (`src/components/Button.js` or wherever the default lives):
- When `variant` is primary (the default), default text color changes from hardcoded `#fff` to `colors.onPrimary`.
- `textColorProp` prop still overrides this, so any intentional overrides continue to work.

**Call-site audit:** grep for `textColorProp="white"` across `src/`. Known sites:
- `AddDeviceScreen.js` (NFC write button area)
- `QuickActionModalScreen.tsx`
- A handful of modal action buttons

Change each to either:
- Remove the prop (lets the new `onPrimary` default take over), or
- Change to `textColorProp={colors.onPrimary}` if the surrounding code pulls `colors` from `useTheme()`.

**Unchanged tokens:** status colors (`success`, `error`, `warning`, `info`), surface colors (`background`, `surface`, `card`, `surfaceAlt`), text colors, border colors. The ~60 screens migrated in the earlier branding commit automatically pick up the new primary via `useTheme().colors`.

## Architecture notes

- No new services, screens, or contexts. Phase 1 is entirely configuration + theme-token + asset + text-label changes.
- The `Button` component edit is the only shared-component change. Everything else is per-call-site or per-file.
- Testing: since Jest's `jest.setup.js` already mocks AsyncStorage, the ThemeContext color changes have no test impact. No new unit tests required. Manual on-device smoke test required for logo rendering and NFC prompt text.

## Risks

- **New bundle ID = new app in stores.** Existing BattWrapz v1.9.5 users don't auto-update to TOOLTRAQ. User has explicitly accepted this.
- **EAS project change.** First build on the new branch requires `eas init` + `eas credentials` manual setup. Failure here would block the first production build but not code merge.
- **Yellow readability in light mode.** Mitigation: `onPrimary` token enforces dark text; manual visual check during on-device test is part of the plan acceptance.
- **Deep-link fallback.** Keeping `wrapbattz://` in `linking.prefixes` means old URLs continue to work for existing installs; no user-visible breakage.

## Out-of-band manual steps (not implemented in code)

1. Run `eas init` on the new branch once (produces new `projectId` / updates URL — commit the resulting `app.json` change as part of the branch).
2. Run `eas credentials` for iOS + Android to register the new bundle ID with App Store Connect and Play Console.
3. Produce square icon / splash variants of the TOOLTRAQ logo at a later date and swap in.
4. After confirming TOOLTRAQ on-device: delete orphan `assets/logo-transparent.png` and `assets/logo.jpg`.
